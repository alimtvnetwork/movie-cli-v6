# Updater RCA — Deploy-path vs PATH mismatch left active binary frozen

**Severity:** High (silent rollback to old binary on every update)
**Fixed in:** v2.121.0
**Files:** `run.ps1` (`Resolve-DeployTarget`, post-deploy PATH-sync block), `updater/cleanup.go`

## Symptom (verbatim from user)

```
🎯 Active binary: E:\bin-run\movie.exe
...
  Version before: movie-cli v2.97.0   Commit: 6287058
...
  OK Deployed to: D:\bin-run\movie.exe
  !! PATH resolves 'movie' to a different binary: E:\bin-run\movie.exe
  !! Active PATH binary is in use; retrying (1/5)...
  !! Could not sync active PATH binary after retries.
movie.exe :   ⚠ Could not remove movie-update-29152.exe: Access is denied.
    + FullyQualifiedErrorId : NativeCommandError
```

The user ran `movie update` repeatedly. Each run successfully built and deployed
a new binary, but the *active* binary on PATH never changed — every next run
still reported `Version before: movie-cli v2.97.0`. The PATH-sync retry loop
then printed five "in use" warnings, and the cleanup pass ended in a red
PowerShell `NativeCommandError` block.

## Root cause

Two independent paths in the user's environment:

| Path           | Source                       | Contained                |
|----------------|------------------------------|--------------------------|
| `D:\bin-run`   | `powershell.json` `deployPath` | newly-built binary       |
| `E:\bin-run`   | `$env:PATH`                   | frozen v2.97.0 binary    |

The handoff worker `movie update` invoked was a copy of `E:\bin-run\movie.exe`
— the v2.97.0 binary. That old worker generated a temp PowerShell handoff
script that called `run.ps1 -Update` **without** `-TargetBinaryPath`. With no
target override, `Deploy-Binary` fell back to `$config.deployPath` (`D:\bin-run`).
The deploy succeeded — but to the *wrong* drive. The PATH binary stayed at
v2.97.0, so the next `movie update` again handed off a v2.97.0 worker, which
again deployed to D:, ad infinitum.

The post-deploy PATH-sync retry loop tried to copy from `D:\bin-run\movie.exe`
over `E:\bin-run\movie.exe`, which is *itself the parent process*. Windows
holds an exclusive lock on a running .exe, so every copy attempt failed with
`Access is denied` and produced the visible retry spam.

The cleanup pass's `os.Remove(movie-update-29152.exe)` failed for the same
reason — that file is the live worker. The error was written to `os.Stderr`,
which PowerShell's native-command pipeline wraps in the red
`NativeCommandError` block the user kept seeing.

## Why earlier fixes did not work

- **v2.118.0** added `-TargetBinaryPath` plumbing — but only **new** workers
  send it. The user's worker was always v2.97.0, so `-TargetBinaryPath` was
  never present.
- **v2.119.0** stopped the *new* worker from running the PATH-sync loop — but
  the user's old worker still ran it because the rendered temp script was
  generated by v2.97.0.
- **v2.120.0** added `Normalize-LegacyUpdateArgs` to recover legacy parameter
  names — but the v2.97.0 worker didn't pass *any* form of target path; it
  passed nothing.

The repo-side fixes were correct in isolation but couldn't take effect because
the active binary never updated to a version that included them.

## Fix (v2.121.0)

Single, repo-side change that works regardless of which old worker calls in:

1. **`Resolve-DeployTarget` now falls back to `Get-Command movie` when
   `-Update` is set and `-TargetBinaryPath` is empty.** The active PATH binary
   becomes the deploy target — `powershell.json`'s `deployPath` is ignored in
   update mode unless explicitly overridden via `-TargetBinaryPath` or
   `-DeployPath`. This guarantees the binary the user is *actually running*
   gets replaced, breaking the loop on the very next `movie update`.
2. **Post-deploy PATH-sync loop is unconditionally skipped in `-Update` mode.**
   The deploy already wrote to the right file; the retry/copy loop only ever
   produced false-alarm "in use" warnings.
3. **`updater/cleanup.go` now silently skips any locked `*-update-*` worker
   leftover** and routes remaining warnings to stdout. PowerShell no longer
   wraps cleanup output in `NativeCommandError`.

## One-time bootstrap

The user must still upgrade past v2.97.0 once. Either:
- Wait for the next `movie update` (now correct — deploys to PATH binary).
- Or run from a fresh shell: `Copy-Item "D:\bin-run\movie.exe" "E:\bin-run\movie.exe" -Force`

## Prevention

- Update mode must never trust `powershell.json`'s `deployPath`. Always derive
  the target from the active PATH binary, with `-TargetBinaryPath` as override.
- `os.Remove` on a worker copy must never write to stderr; permission errors
  on `*-update-*` artifacts are expected and self-healing on the next run.
